# Terraform backend bootstrap

`bootstrap-backend.sh` creates the remote state backend shared by the
`networking`, `compute` and `services` layers, and generates the local
config files each layer needs to use it. It's the first thing to run before
applying any layer. `destroy-backend.sh` is its counterpart: it tears the
backend back down.

## What it creates in AWS

- **S3 bucket** `<project>-<environment>-terraform-state-<account-id>`
  - Versioning enabled.
  - Server-side encryption (AES256).
  - All public access blocked.
- **DynamoDB table** `<project>-<environment>-terraform-locks`
  - `PAY_PER_REQUEST` billing (no fixed cost).
  - `LockID` as the partition key, used by Terraform to lock the state
    during `plan`/`apply` and avoid concurrent runs stepping on each other.

Both are idempotent: if they already exist, the script reuses them instead
of failing.

## What it creates locally

These files are gitignored (see `infra/.gitignore`) since they're
environment-specific and derived, not shared config:

- `networking/backend.<environment>.hcl`
- `compute/backend.<environment>.hcl`
- `services/backend.<environment>.hcl`

  Each points at the same bucket/table, with a different state `key`
  (`<environment>/<layer>/terraform.tfstate`) so every layer keeps its own
  state file inside the shared bucket.

- `compute/backend.auto.tfvars`: passes the state bucket name and the
  `networking` layer's state key into `compute`, which reads `networking`'s
  outputs (VPC id, subnet ids) via `terraform_remote_state`.

## Tearing it down

⚠️ **`destroy-backend.sh` is destructive**: it permanently deletes the S3
bucket — including every version of every layer's `terraform.tfstate` — and
the DynamoDB lock table. It asks you to type the bucket name back as
confirmation before doing anything.

Deleting the bucket does **not** destroy the AWS resources described by
those states (the VPC, the EKS cluster, the ECR repos, ...) — it just makes
them unmanageable by Terraform afterwards. Always run `terraform destroy`
on every layer (`networking`, `compute`, `services`) first; only tear down
the backend itself once nothing needs that state anymore.

It also removes the local `backend.<environment>.hcl` /
`backend.auto.tfvars` files generated by `bootstrap-backend.sh`, since
they'd otherwise point at a backend that no longer exists.

## Usage

Requires AWS credentials with permissions for S3 and DynamoDB, and the AWS
CLI installed.

Via the Makefile (recommended):

```bash
cd infra/backend
make create    # bootstrap-backend.sh
make destroy   # destroy-backend.sh (asks for confirmation)
```

Or directly:

```bash
cd infra/backend
./bootstrap-backend.sh
./destroy-backend.sh
```

Override the defaults with environment variables (same names for either
script or `make` target):

```bash
PROJECT_NAME=modern-ai-strategies ENVIRONMENT=dev AWS_REGION=us-east-1 ./bootstrap-backend.sh
# or
make create ENVIRONMENT=prod
```

`bootstrap-backend.sh` is safe to run more than once — existing resources
are detected and left as they are; only the local `backend.*.hcl` /
`backend.auto.tfvars` files are regenerated.
